# Slang Pointer Codegen: The Phase 6 Design

Companion note to [../vulkan_1_3_migration.md](../vulkan_1_3_migration.md). **Status: Steps A and B done (2026-07-16); Steps C–F pending. Step A's finding fired the D2 contingency — default `T*` pointee layout is *natural*, not std430 — and D1/D2/D6 plus the Step D sketch were re-planned accordingly (see D2 for the resolved design).** Phase 6 is build-time only: reflection → JSON → Rust codegen. The runtime plumbing (`Gpu::device_address` etc.) landed in Phase 5 — see [bda_renderer_plumbing.md](bda_renderer_plumbing.md) — and the first real shader consumer is Phase 7 (`sprite_batch` + `particles`).

## The concept

A `T*` field in a Slang ParameterBlock is 8 bytes of uniform data. Slang reports it as `ParameterCategory::Uniform`, so `param_binding` (src/shaders/reflection/parameters.rs:429–461) already yields `Binding::Uniform { offset, size: 8 }` with zero changes. It consumes no descriptor slot: descriptor ranges are collected in a separate reflection path (src/shaders/reflection/pipeline_layout.rs, driven by `descriptor_set_descriptor_range_*` queries), and a pointer's 8 bytes are simply absorbed into the automatically-introduced uniform buffer's `size(Uniform)`. **Descriptor set layouts, `create_descriptor_sets` (renderer.rs:3681), `Resources<'a>`, and `pipeline_config()` need no changes at all.**

So the whole phase reduces to: teach the struct-field walk a `Pointer` kind, emit `pub name: u64` at the reflected offset, and emit the pointee's Rust struct the same way `StructuredBuffer<T>` element structs are emitted today.

The danger is that pointee layout under `PhysicalStorageBuffer64` is a *silent* contract — no validation layer checks loads through a raw device address, so a layout mismatch is just quietly wrong data. The plan therefore front-loads an empirical layout experiment (Step A) before any Rust depends on the answer, and hardens everything with per-field `offset_of!` asserts enforced at `cargo check` time (both in the main crate and in `shaders/test/check_crate`, which `alignment_tests` compiles — build_tasks.rs:1434–1514).

## Discovered while planning: the Mat3 bug is live, not dormant

The parent doc calls the `float2x2/3x3 → glam::Mat2/Mat3` mapping (build_tasks.rs:989–999) a *dormant* bug. It is actually **live in a test shader**: `shaders/test/std430_matrices.shader.slang` puts `float2x2 a; float3x3 b;` in a StructuredBuffer element, and the committed snapshot emits `pub b: glam::Mat3` — 36 contiguous bytes where the GPU expects 48 with 16-byte column stride. The total-size assert passes only by coincidence (the following `Mat4`'s align-16 padding absorbs the 12-byte difference), and per-field `offset_of!` asserts would *not* catch it either — the surrounding field offsets coincide; the error is *interior column stride*.

Two consequences baked into this design:

- The decided matrix hard-error (Step B) must also fix `std430_matrices.shader.slang` (drop the mat2/mat3 fields, mirroring the explanatory comment `std140_matrices.shader.slang` already carries).
- **Invariant: contiguous-exact Rust types only.** The generator may only emit Rust types whose byte image exactly matches the GPU bytes with no interior stride: scalars, `u64`, `glam::Vec2/Vec3/Vec4/Mat4`, `[u8; N]` padding. `offset_of!` asserts cannot police interior stride, but per-field **size** checks can — stride padding always changes a field's total size (`size_of::<glam::Mat3>()` = 36 vs reflected 48). Step B lands that check as a unit test (red on the live bug, green after the fix); Step C promotes it into generated `const` asserts. The matrix hard error removes the only current violation.

## The design

### D1 — JSON shape for pointer fields

Add `Pointer(PointerStructField)` to `StructField` (src/shaders/json/parameters.rs:82–90, serde `tag = "kind"`, camelCase):

```rust
#[derive(Debug, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct PointerStructField {
    pub field_name: String,
    pub binding: Binding,               // always Uniform { offset, size: 8 }
    pub pointee_type: StructFieldType,  // reuse existing { type_name, fields }
    pub pointee_size: usize,            // reflected size(Uniform) of the pointee
}
```

- **Struct pointees only** for now; reflection bails clearly on scalar/vector pointees. Reusing `StructFieldType` (the carrier `StructStructField` already uses) makes the codegen recursion shape identical to the existing StructuredBuffer-of-struct path. JSON files are wholly regenerated by `just shaders`, so extending later is free.
- **`pointee_type.fields` (and `pointee_size`) must be reflected from `program_layout.type_layout(pointee_ty, LayoutRules::DefaultStructuredBuffer)`, not from the pointer's `element_type_layout()`** — the latter reports the wrong (natural) offsets for `Std430DataLayout` pointers; see D2 finding 3.
- `pointee_size` is a belt-and-suspenders cross-check: codegen asserts its own computed struct size equals what Slang reflected, catching divergence in trailing-pad math.
- **Struct-def gathering/dedup:** the `gather_struct_defs` Pointer arm mirrors the `ResourceResultType::Struct` arm (build_tasks.rs:~876–897): `generate_std430_struct_fields(&ptr.pointee_type.fields, struct_defs)` then `try_add_struct_def`. A struct referenced both by a pointer and a `StructuredBuffer` produces byte-identical defs and dedups. Shared modules need nothing new (`tag_source_modules` works on type names).
- **Gap to close while here:** `collect_shared_modules` (build_tasks.rs:~1291–1309) dedups by type name with *no* field-compatibility check — first definition silently wins. Factor the comparison out of `try_add_struct_def` and apply it there too, so a cross-shader layout divergence panics instead of silently dropping one layout.

### D2 — Pointee layout: settled empirically (Step A, 2026-07-16)

The expectation was std430; the experiment (layout-hostile pointee `HostileData`, compiled through the real session setup, cross-checking reflection against rspirv-disassembled SPIR-V) found the contingency case instead. Four findings, all verified on vendored slang v2026.13.1:

1. **Default `T*` pointee layout is *natural* (C-like) layout, not std430.** SPIR-V: struct literally named `HostileData_natural`, offsets 0/12/16/**20**/36/52/56/68, `ArrayStride 72`, loads `Aligned 4` (`float4 d` at 20; `InnerA.v` at 4). The pointer's reflected `element_type_layout()` agrees exactly (size 72, align 4) — reflection ⇔ SPIR-V consistent for the default case.
2. **Slang has per-pointer layout types**: `Ptr<T, Access, AddressSpace, L: IBufferDataLayout = DefaultDataLayout>` with `Std430DataLayout`/`Std140DataLayout`/`ScalarDataLayout`/`CDataLayout` (slang/source/slang/hlsl.meta.slang:24–72) and the alias `LayoutPtr<T, L>` (core.meta.slang:1512). `LayoutPtr<HostileData, Std430DataLayout>` emits **true std430 SPIR-V**: offsets 0/12/16/32/48/80/88/104, `ArrayStride 112` — exactly the D6 predictions.
3. **But the pointer's `element_type_layout()` lies for non-default layouts**: for the `Std430DataLayout` pointer it still reports the natural offsets (72/20/36). The correct std430 offsets ARE obtainable from reflection via `program_layout.type_layout(pointee_ty, LayoutRules::DefaultStructuredBuffer)` (fork API `Shader::type_layout`, shader.rs:90) — verified to return size 112, offsets 0/12/16/32/48/80/88/104, matching the SPIR-V byte-for-byte. (`LayoutRules::Default`/`DefaultConstantBuffer` return the std140 layout, size 128.)
4. **The declared layout is detectable**: `Type::full_name()` on the pointer field's type returns `Ptr<HostileData, Access.ReadWrite, AddressSpace.Device, DefaultDataLayout>` vs `…, Std430DataLayout>`. (Caveat: `Type::element_type()` on the pointer type returns None — get the pointee `Type` via `element_type_layout().ty()`.)

Also confirmed en route: the pointer field itself is `ParameterCategory::Uniform`, offset 8, size 8 in the std140 param block (the `Binding::Uniform` assumption holds); SPIR-V is 1.5 with auto-enabled `PhysicalStorageBufferAddresses` + `SPV_KHR_physical_storage_buffer` (no compiler-option change needed); and there is no global flag that forces std430 for pointers only (`-fvk-use-scalar-layout`/`-fvk-use-c-layout` exist but change ConstantBuffer/StructuredBuffer layouts too).

**Resolved design — require std430-layout pointers:**

- Shaders must declare pointer fields as `Ptr<T, Access.ReadWrite, AddressSpace.Device, Std430DataLayout>`; Step E adds a project `typealias` in a shared module (e.g. `BufferPtr<T>`) so shaders write `BufferPtr<Sprite> sprites;`.
- Reflection parses the pointer type's `full_name()`; a `DefaultDataLayout` (or any non-`Std430DataLayout`) pointer is a **hard error** whose message names the alias — never silently accepted, because its natural layout would otherwise be codegen'd as std430 and silently corrupt (finding 3 means the pointee's own reflected layout cannot be trusted to reveal the mismatch).
- Pointee field offsets come from `program_layout.type_layout(pointee_ty, LayoutRules::DefaultStructuredBuffer)`, NOT from the pointer's `element_type_layout()`. This stays reflection-driven (no local layout table) and matches the emitted SPIR-V. Implementation note: `reflect_struct_fields` doesn't currently see the program layout — the Pointer arm needs the `slang::reflection::Shader` threaded down (or the pointee resolved before recursion).
- Everything downstream keeps the original std430 shape: reuse `Alignment::Std430` + `generate_std430_struct_fields`; dual-context types (same struct via `BufferPtr<T>` and `StructuredBuffer<T>`) are byte-identical **by construction**; glam types stay usable (std430 never mis-aligns them); no new device feature needed (natural layout would have required `scalarBlockLayout`).
- Slang-upgrade drift (default layout changing, `full_name` format changing, `DefaultStructuredBuffer` ≠ std430) is pinned by Step E's permanent rspirv offset test plus the `pointee_size` cross-check.

Rejected alternative — embrace natural-layout pointees (default `T*`): reflection agrees with SPIR-V there, but it forfeits dual-context byte-identity (natural ≠ std430 whenever a `float3`/`float4` needs padding), forces `[f32; N]` fallbacks where glam types land mis-aligned (e.g. `float4` at offset 20), and requires enabling the `scalarBlockLayout` device feature for the SPIR-V to be valid. Std430-required is strictly simpler and keeps Phase 7's dual-context migration story.

### D3 — Per-field `offset_of!` asserts on ALL GPU structs

Emit for every GPU-layout struct, not just pointer pointees: the descriptor path has the same silent-failure class, the churn is additive-only and mechanically reviewable, and landing it *before* pointer support keeps the pointer steps' zero-churn gates meaningful.

Shape: `GeneratedStructFieldDefinition` (build_tasks.rs:1091–1111) gains `offset: Option<usize>` (`new()`/`padding()` → `None`); `generate_std430_struct_fields` (732–789) and `generate_std140_struct_fields` (794–854) set it where they already consult `field_offset_size`. Vertex structs and `Resources<'a>` never get offsets, so asserts appear exactly on GPU-layout structs with no extra gating. Template block added after the existing size assert in all three templates (shader_atlas_entry.rs.askama ~44, shader_compute_entry.rs.askama ~41, shader_shared_module.rs.askama ~34 — identical struct blocks): for each field with `Some(offset)`, emit

```rust
const _: () = assert!(core::mem::offset_of!(TypeName, field_name) == N);
```

### D4 — Nested pointers: error initially

`reflect_struct_fields` gains an `in_pointer_pointee: bool` parameter; the Pointer arm bails if already inside a pointee, *before* recursing. This is not just scope-trimming — it's the cycle guard (`struct Node { Node* next; }` would recurse infinitely during reflection). Descending through a Resource element keeps `false`, so a `StructuredBuffer<T>` whose `T` contains a pointer still works one level deep. Lifting later = drop the bail + add a visited-type-name stack.

### D5 — glam guards

- Padding stays `[u8; N]` (align 1) — already true.
- Reflected offset not a multiple of the Rust type's alignment → **hard error** (panic in `generate_*_struct_fields`), not an `[f32; N]` fallback — it's unreachable under std140/std430, so a fallback is dead complexity. Requires a new `rust_type_alignment` map, distinct from the *GPU* `field_alignment` table (build_tasks.rs:1172–1182): `glam::Vec4`/`glam::Mat4` → 16, `glam::Vec2` → 8, `glam::Vec3` → **4** (this is what lets fields tail-pack after a `float3` — never substitute `Vec3A`), `f32`/`u32` → 4, `u64` → 8; generated struct types skip the check (their `repr(align)` equals their GPU alignment, which divides their reflected offset).
- Emit `const _: () = assert!(std::mem::align_of::<glam::Vec4>() == 16);` once per generated file (all three templates) — catches a transitively-enabled glam `scalar-math` feature at `cargo check`, in both the main crate and check_crate.

### D6 — Test shaders and pin tests

Three new files in `shaders/test/` (auto-discovered by `alignment_tests`):

All pointer declarations in these shaders use the std430-layout form per D2 — `BufferPtr<T>` (the project alias for `Ptr<T, Access.ReadWrite, AddressSpace.Device, Std430DataLayout>`), never bare `T*`.

1. **`pointer_pointee_layout.shader.slang`** — the layout-hostile pointee, the *same* struct as Step A's experiment so the committed test pins the finding. The param block also exercises pointer *placement*: `float scale; BufferPtr<HostileData> items; float2 post;` (pointer at std140 offset 8, `post` at 16). Pointee, with expected std430 offsets (verified against SPIR-V in Step A):

```slang
struct InnerA { float x; float3 v; }   // std430 size 32 (v at 16); scalar layout would put v at 4
struct InnerB { float2 p; float q; }   // std430 align 8; std140 would align it 16

struct HostileData {
    float3 a;    // 0
    float  b;    // 12  (tail-packs into float3's pad)
    float  c;    // 16  (lone float before a float4)
    float4 d;    // 32
    InnerA ia;   // 48
    float  e;    // 80
    InnerB ib;   // 88  (std140 would say 96)
    float  f;    // 104
}                // total size 112
```

`InnerA` discriminates std-vs-scalar layout; `InnerB` discriminates std430-vs-std140. The vertex shader reads `params.items[id]` so an `ArrayStride`/`PtrAccessChain` appears in the SPIR-V.

2. **`pointer_dual_context.shader.slang`** — one struct (`DualData { float3 pos; float w; float4 color; }`) used as **both** `BufferPtr<DualData> by_pointer` and `StructuredBuffer<DualData> by_buffer` in one param block. Pins the dual-context invariant mechanically: if pointer layout ever diverges from std430, `try_add_struct_def`'s compatibility check panics inside `alignment_tests`.

3. **`pointer_params.compute.slang`** — minimal compute (`struct Params { BufferPtr<PtrItem> data; uint count; }`, `PtrItem { float2 a; float b; }`). `shaders/test/` currently has zero compute shaders, so this is what makes check_crate actually compile the compute template's new assert block.

Plus three unit tests in build_tasks.rs `tests` (a fourth — the field-size tripwire — lands earlier, in Step B):

- **Permanent SPIR-V pin test**: compile `pointer_pointee_layout.shader.slang` via `prepare_reflected_shader`, parse the vertex SPIR-V with the already-in-use `rspirv` crate (cf. `count_branch_instructions`, build_tasks.rs:1516), locate the PhysicalStorageBuffer-storage-class struct, assert its `OpMemberDecorate Offset`s (0/12/16/32/48/80/88/104) and the pointer `ArrayStride` (112) equal the std430 values Step A verified. This is the regression guard for future slang upgrades.
- **Default-layout-pointer rejection test**: a shader (compiled from an inline string or scratch fixture, NOT a file in `shaders/test/` — it must fail) declaring a bare `T*` field, asserting reflection errors with the message naming `BufferPtr`/`Std430DataLayout`. This pins D2's hard error — the one guard standing between a bare `T*` and silent natural-vs-std430 corruption.
- **`#[should_panic]` matrix-error test** calling `gather_struct_defs` with a synthetic `MatrixStructField` — the hard error can't live as a test shader (it would fail the whole `alignment_tests` compile pass).

## Rules of thumb (gotchas encoded in this design)

- **BDA layout failures are silent.** No validation layer checks loads through a device address; the defense is layered tripwires, not runtime checks — the rspirv pin test, the `pointee_size` cross-check, the dual-context shader, and the `offset_of!` asserts fail at `just test`/`cargo check` time instead.
- **`offset_of!` cannot see interior stride.** A field's surrounding offsets can coincide while its interior layout is wrong (the live Mat3 bug). Per-field *size* checks close most of the gap — stride padding always changes total size — via the Step B unit test and the Step C generated `size_of` asserts. The uncovered residue (same-size interior rearrangement) doesn't exist in this type universe; don't add a Rust type to the generator without checking its byte image is stride-free.
- **A pointer is uniform data, not a resource.** If a change makes pointers show up in `Resources`, descriptor layouts, or `pipeline_config()`, something is wrong upstream.
- **Never trust a pointer's `element_type_layout()` for offsets.** It reports the *default* (natural) layout regardless of the pointer's declared `IBufferDataLayout` (Step A finding 3). Pointee layout always comes from `program_layout.type_layout(pointee_ty, LayoutRules::DefaultStructuredBuffer)`, and only `Std430DataLayout` pointers are accepted.
- **Bare `T*` must stay a hard error.** Its natural layout means glam-hostile offsets and dual-context divergence; if natural-layout support is ever wanted, that's a deliberate design change (new `Alignment` variant, `scalarBlockLayout` device feature, `[f32; N]` fallbacks), not a relaxation of the check.
- **Never substitute `Vec3A` for `Vec3`.** glam's align-4 `Vec3` is what allows tail-packing after a `float3` (e.g. `Sprite.rotation` at offset 12).
- **Churn review must be mechanical.** Step C touches ~40 snapshots; the gate is "every changed line matches the assert/guard patterns", checked by script — an eyeballed `cargo insta accept` could let a real layout diff through.
